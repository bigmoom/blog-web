散列表

散列表的基础知识

散列表是一种非常常见的数据结构，最大的优点就是插入，删除和查找一个元素都只需要\(O(1)\)的时间复杂度，所以哈希表通常被用来以空间换时间，优化时间效率。而哈希的基本思想就是散列，即将不同的关键字映射到不同的存储单元中。

散列函数

散列函数即hashCode()，用来不同的关键字如何进行映射。

例如简单的对整数进行散列，即取模计算 \[ key\space mode \space tableSize \]

不同情况我们可以使用不同的散列函数，所以散列函数各不相同，我们简单看下Integer和String类型的hashCode()

  • Integer ：直接定址法，即散列值为原本的值

  • String：根据拉丁编码和UTF16编码进行了区分，

我们重点看一下HashMap中的散列函数，其实叫做扰动函数更合适

HashMap的hash()其实是先拿到key的hashCode，是一个32位的int类型的值，然后让hashCode的高16位和低16位进行异或操作，这样可以有效降低冲突概率

因为 key.hashCode() 函数调用的是 key 键值类型自带的哈希函数，返回 int 型散列值。int 值范围为-2147483648~2147483647，加起来大概 40 亿的映射空间。只要哈希函数映射得比较均匀松散，一般应用是很难出现碰撞的。但问题是一个 40 亿长度的数组，内存是放不下的。

我们先看一下HashMap是如何计算数组index的：

也是使用了一个模运算，就是把散列值和数组长度-1做了一个与&操作

位运算比取余运算要快

以数组长度16为例，16-1=15，二进制表示为1111，那么与某个散列值做与操作：

其实也就是截取了最低的四位，高位信息完全被抛弃，这样的话碰撞也会非常严重

我们再看有了扰动函数之后：

右移十六位，将自己的高位半区和低位半区做异或操作，这样即混合了原始散列值的高位和低位信息，即增加了低位的随机性，也使得低位能够掺杂高位信息

散列冲突

不同key的散列值相同即散列冲突是非常常见的现象，也是我们需要重点需要解决的问题。

分离链接法

如上图所示，散列表的值指向的是一个链表，链表中存储着相同散列值的元素，每当插入出现冲突的时候，将新的值插入到链表头部。这样对于查询操作，每次O(1)的时间复杂度查找到链表的位置，然后最差O(N)的时间复杂度遍历链表。

插入到头部是因为一般新放进去的元素大概率会被经常使用

jdk1.8后又改成了尾插法，即新元素插入到链表尾部

可以发现，当链表过长的时候，我们的查询效率其实不会很好，所以可以将链表转化为红黑树，具体的红黑树方面的操作之前已经提过，暂时不作赘述

我们这里以JAVA中的HashMap为例

可以看到树化的条件是数组长度大于64，链表长度大于8，对于数组长度涉及到扩容，我们后面会具体讲到，这边先看链表长度为8

为什么选8作为树化的条件，源码中也给我们做了解释

大概意思就是，根据大量实验发现，hash冲突发生8次的概率极低，只有0.00000006，几乎为不可能时间，那么如果真的发生了8次，那么说明由于元素或者散列函数的设计问题，发生冲突的概率很大，后序还可能发生冲突，所以应该将链表转化为红黑树

那么为什么红黑树转链表的阈值是6呢，因为如果将阈值也设置为8，那么会出现红黑树和链表之间不停转换的情况，浪费系统资源。

线性探测法

简单的说就是发生冲突之后，从冲突的位置开始依次判断下一个位置是否空闲，直到找到空闲位置

可以发现，这种情况即使在表较空的情况也可能形成一些区块，即造成一次聚集，使得插入和查询等操作效率大大降低

平方探测法

平方探测其实就是优化线性探测中一次聚集问题，即从冲突的位置x开始，第一次增加1^2个位置，第二次增加2^2…，直至找到空闲的位置

虽然平方探测法，但是由于散列到统一位置的那些元素，他们的备选单元都相同，所以可能导致无法插入等问题。这就是二次聚集问题

双散列

所谓双重散列，意思就是不仅要使用一个散列函数。我们使用一组散列函数 hash1(key)，hash2(key)，hash3(key)……我们先用第一个散列函数，如果计算得到的存储位置已经被占用，再用第二个散列函数，依次类推，直到找到空闲的存储位置。

其他方法

还有很多解决散列冲突的方法，例如建立公共溢出区，布谷鸟散列，完美散列，跳房子散列等等，有空再补上

扩容

我们上面讨论的都是在散列表的数组长度不变的情况下考虑解决冲突，但是当散列表中元素达到一定量的时候其实冲突的概率就普遍很大了，我们更需要的是对散列表进行扩容，即提供更多的空间进行散列

再散列其实也就是扩容操作解决冲突问题

装载因子（load factor）

设散列表的大小为\(m\)，填入表中的元素个数是\(n\)，则装载因子为 \[ \lambda = \frac{n}{m} \] 常见的散列表中装载因子一般设置为0.5到0.8为宜，HashMap中装载因子默认为0.75

初始容量

可以看到HashMap初始化的时候调用了tableSizeFor这个方法，看起来挺复杂，其实就是找到比initialCapacity大的最小的2的倍数

initialCapacity不是2的次幂值的情况

例如\(n=17\)，二进制为1000

那么我右移一位后0100，与操作之后为1100

右移两位后0011，与操作之后1111，后面的更多位我们暂时用不到

最后再加1，即容量为32,

我们第一部n=cap-1，其实就是为了处理cap本身就是2的次幂的情况，这种情况直接作为容量，不需要再找比他大的容量

那么我们为什么设置HashMap初始值为2的次幂值呢？

  • 方便哈希取余

    我们上面介绍过了HashMap采用的是取模操作

    这里能达到和取模效果相同主要就是因为他的容量为2的次幂值，2的次幂值即意味着二进制位上只有1位是1，所以该数-1即可使得二进制位上1变为0，其他的变成1，再通过&运算就可以达到和%一样的操作，并且位运算比取模运算效率要高得多

    此外，由于n-1的二进制全是1，这样与添加元素的散列值进行&运算就可以使得散列充分且均匀，减少冲突

  • 扩容方便

    扩容之后的大小也是2的次幂值，可以把元素完美的转移到新的散列表中

扩容机制

为了减少冲突的概率，当表中元素到达一定个数的时候，就会触发扩容机制，即把所有的元素rehash后再放到扩容之后的散列表中

例如我们容器大小为16，装填因子是0.75，那么\(16*0.75=12\)，即当元素个数超过12个时就要进行扩容

具体扩容机制不同的Hash类型不同，这里我们以HashMap为例

因为HashMap的初始容量是2的次幂值，扩容之后长度为原来的两倍，依然是2的次幂值，所以元素要么在原位置，要么在原位置再移动2的次幂

如上图所示，key1,key2本来是两个发生冲突的元素，但是通过扩容之后，key2的位置变成了乘2的位置，解决了冲突问题

所以扩容的时候只需要看原来hash值新增的那位是0还是1，是0的话则保持不变，是1的话则变成原来的索引加上扩容前的容量大小

HashMap的实现

hash(),resize()等基础操作还是模拟HashMap，但是没有实现树化操作，只是单纯用链表来处理冲突问题

插入、删除和随机访问都是\(O(1)\)的容器

LeetCode 380

实现RandomizedSet 类：

  • RandomizedSet() 初始化 RandomizedSet 对象
  • bool insert(int val) 当元素 val 不存在时，向集合中插入该项，并返回 true ；否则，返回 false 。
  • bool remove(int val) 当元素 val 存在时，从集合中移除该项，并返回 true ；否则，返回 false 。
  • int getRandom() 随机返回现有集合中的一项（测试用例保证调用此方法时集合中至少存在一个元素）。每个元素应该有 相同的概率 被返回。

你必须实现类的所有函数，并满足每个函数的 平均 时间复杂度为 O(1) 。

我们看到O(1)复杂度其实可以直接联想到散列表，但是题目要求随机访问也是O(1)，散列表无法实现随机访问，所以我们使用一个数组存储数组，然后使用一个散列表记录元素在数组中的index即可

最近最少使用缓存

LeetCode 146

请你设计并实现一个满足 LRU (最近最少使用) 缓存 约束的数据结构。

实现 LRUCache 类：

  • LRUCache(int capacity) 以 正整数 作为容量 capacity 初始化 LRU 缓存
  • int get(int key) 如果关键字 key 存在于缓存中，则返回关键字的值，否则返回 -1 。
  • void put(int key, int value) 如果关键字 key 已经存在，则变更其数据值 value ；如果不存在，则向缓存中插入该组 key-value 。如果插入操作导致关键字数量超过 capacity ，则应该 逐出 最久未使用的关键字。

函数 get 和 put 必须以 O(1) 的平均时间复杂度运行。

题目中\(O(1)\)时间复杂说明肯定使用散列表来实现，但是需要进行调整以实现LRU的功能

为了保证记录最久未使用的项，我们可以使用一个链表来实现这个功能

其实JAVA有已经实现好的链表散列表LinkedHashMap，但是一般还是要手写的

我们设置头节点和尾结点两个哨兵节点，链表第一项放最久未使用的，插入新元素的时候再末尾依次插入，每当有更新则把该结点移到最后即可。然后当容量已满的时候，只需要删除第一个结点即可

散列表的应用

因为散列表的插入删除查询时间复杂度都是\(O(1)\)，所以是非常常用的数据结构

当散列表的键取值范围固定时我们也可以通过数组模拟散列表，更加简洁，例如记录二十六个字母出现的次数

有效的变位词

给定两个字符串 *s* 和 *t* ，编写一个函数来判断 *t* 是否是 *s* 的字母异位词。

注意：若 *s* 和 *t* 中每个字符出现的次数都相同，则称 *s* 和 *t* 互为字母异位词。

这套题目非常简单，主要是如果是单纯的只有小写字母，我们可以使用数组来统计次数

但是如果不仅仅是字母，更可能还有其他的字符，例如Unicode编码的字符，那么我们就直接使用HashMap来存储

变位词组

LeetCode 49

给你一个字符串数组，请你将 字母异位词 组合在一起。可以按任意顺序返回结果列表。

字母异位词 是由重新排列源单词的字母得到的一个新单词，所有源单词中的字母通常恰好只用一次。

这道题我们可以把每个字符串排序，因为字母异位词排好序之后都是一样的，例如eat，tea排序之后都是aet，所以我们可以以排好序的字符串作为散列表的键，字母异位词作为值放进去。

还有一种巧妙的解法，但是不太完美

我们中心思路是要为每个字母异位词找到唯一的值对应，所以我们用质数的乘积来计算这个唯一值

一个数如果能拆分成质数的积，那么该组质数唯一

外星语言是否排序

LeetCode 953

某种外星语也使用英文小写字母，但可能顺序 order 不同。字母表的顺序（order）是一些小写字母的排列。

给定一组用外星语书写的单词 words，以及其字母表的顺序 order，只有当给定的单词在这种外星语中按字典序排列时，返回 true；否则，返回 false。

这道题其实很简单，只需要用一个散列表记住每个字母的顺序，然后比较每个字母是否按序即可

最小时间差

LeetCode 539

给定一个 24 小时制（小时:分钟 "HH:MM"）的时间列表，找出列表中任意两个时间的最小时间差并以分钟数表示。

这道题因为是时间序列，我们可以建立一个24*60=1440长度的数组，即每一个位表示一个时间戳，遍历输入，将时间数组中对应的时间位置设置为true，然后比较连续两个true的时间戳的差值即可，这里我们需要注意一个问题，例如上例中22点和0点之间的差值不应该是22而是把0点当做第二天的0点，即0+24，所以最小时间差是24-22=2小时
